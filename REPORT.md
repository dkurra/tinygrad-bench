# tinygrad-bench: an unsaturated coding-agent benchmark from tinygrad

## Summary

I converted [tinygrad/tinygrad](https://github.com/tinygrad/tinygrad) — a real-world
deep-learning framework with **11,879 merged PRs** — into a Docker-based evaluation
benchmark of **100 automatically mined, automatically verified, automatically scored**
SWE-bench-style tasks, and evaluated **5 models** (3 full runs, 2 labeled partials)
across 5 model families with mini-swe-agent under identical conditions.

Everything is explorable via the bundled CLI: `./bench leaderboard`,
`./bench inspect random` (task cards), `./bench traj glm <instance>` (step-by-step
agent-trajectory replay), `./bench run <any-litellm-model>` (evaluate a new model with
one command).

Why tinygrad: it is a genuine ML/AI codebase (autograd, tensors, JIT compilation,
schedulers, GPU runtimes) yet nearly dependency-free and pure Python, so any historical
commit installs in seconds and its test suite runs CPU-only inside a container — the
properties that make a benchmark cheap, reproducible, and reliable to score. It is also
genuinely hard (dense, idiosyncratic code, minimal comments), which keeps the benchmark
unsaturated.

## Evaluation environment

**One Docker image, per-task state.** `environment/Dockerfile.base` builds
`tinygrad-bench:base` (python:3.11-slim + clang + a full-history tinygrad clone + pinned
test dependencies). Instead of 100 per-task images, every consumer (agent run, task
validation, scoring) starts a fresh container from this image and runs the same setup
command:

```
git checkout -q <base_commit> && rm -rf .git && git init -q && git add -A \
  && git commit -qm 'base state' && pip install -q -e . --no-deps --no-build-isolation
```

Two properties matter here:

- **Anti-leak:** `rm -rf .git` + re-init destroys history, so an agent cannot recover the
  gold fix from `git log`/`git show` (the clone contains the future).
- **Offline:** all containers run with `--network none`, so an agent cannot look up the
  real PR on GitHub. `pip install --no-deps --no-build-isolation` works without network.

Setup takes ~5-10 s per container; the resolved dependency set is identical across all
tasks (task installs are `--no-deps`), so environments are deterministic.

## Task design

**Source: merged PRs, mined locally.** tinygrad squash-merges every PR with a
`title (#N)` subject, so first-parent history enumerates merged PRs and `git diff`
reconstructs each PR's change without API calls. `taskgen/mine_prs.py` scanned 8,707
merged PRs (since 2024-07) and kept 400 candidates that (a) change both production code
(`tinygrad/**`) and tests (`test/**`), (b) fall in a 2-400 changed-code-lines band, ≤8
code files, ≥3 added test lines, (c) don't touch packaging files, and (d) aren't
reverts/bumps/CI-only changes. Each candidate stores `base_commit` (parent of the squash
commit), `patch` (code part of the diff), and `test_patch` (test part).

**Verification: execution, not heuristics.** `taskgen/validate_tasks.py` runs each
candidate in a fresh container: pass **A** = base + test_patch; pass **B** = base + both
patches; pass **C** = repeat of B (flake detector). A task is verified iff
`FAIL_TO_PASS := passed(B) ∩ passed(C) − passed(A)` is non-empty;
`PASS_TO_PASS := passed(A) ∩ passed(B) ∩ passed(C)`. This automatically rejects
refactors whose new tests already pass at base, architecture-specific tests that skip in
the container, tasks whose environment can't be reproduced, tests that flake, and
anything too slow.

Funnel: **8,707 merged PRs → 400 candidates → 386 executed → 274
verified → 100 final tasks** (stratified round-robin over module × difficulty so no
subsystem dominates; difficulty tiers by changed code lines: easy ≤25, medium 26-100,
hard >100).

**Problem statements, leakage-aware.** Agents receive a GitHub-issue-style statement,
generated by Gemini from the PR title/body, linked-issue text, and both diffs *as
context*, under instructions to precisely specify observable behavior and any new/changed
public API semantics (so the hidden tests are satisfiable) while never revealing fix
locations, diff content, or test names. Statements from PRs with real linked issues are
grounded in the original issue text.

## Scoring

`evaluation/score.py` maps a model patch to a score in [0, 1]:

```
score = (FAIL_TO_PASS passed / total) × (PASS_TO_PASS passed / total)
```

computed in a fresh offline container (setup → apply model patch with test-file hunks
stripped → apply gold test patch → single pytest run over the task's test files). Empty
or non-applying patches score 0; partial fixes get partial credit; regressions on
previously-passing tests multiply the score down. `resolved` (score = 1.0) is reported
as the strict SWE-bench-comparable metric.

**Sanity pass:** the gold patch scores 1.0 and the empty patch scores 0.0 on every task
by construction (`score.py --gold` / `--empty`): gold 100/100 scored exactly 1.0; empty 100/100 scored 0.0.

## Benchmark runs

Runner: **mini-swe-agent 2.4.5** batch mode (`mini-extra swebench`) with a custom run
config (`evaluation/tinygrad.yaml`): offline docker containers per instance, the
startup command above, 120 s command timeout, and an explicit step-budget warning in the
instance prompt.

All agents share identical conditions: same prompts, same 40-step limit (unless noted),
same offline containers, same scorer. Models evaluated (via OpenRouter unless noted):

| run | model | tasks | notes |
|---|---|---|---|
| glm-5.2 | `z-ai/glm-5.2` | 100 | frontier-class; $0.93/$3.00 per Mtok |
| deepseek-v4-flash | `deepseek/deepseek-v4-flash` | 100 | $0.09/$0.18 |
| qwen3-coder-30b | `qwen/qwen3-coder-30b-a3b-instruct` | 100 | $0.07/$0.27 |
| minimax-m2.5 | `minimax/minimax-m2.5` | 29 (partial) | $0.15/$0.90; stopped early to reallocate budget/time to glm-5.2 |
| flash-lite | `gemini/gemini-3.1-flash-lite`, `reasoning_effort=high` | 8 (partial) | free-tier Gemini key (~1k requests/day made a full run a multi-day affair); 7 further runs lost to rate-limit exhaustion are excluded |

Two runner paths exist: `evaluation/run_benchmark.sh <litellm-model>` (one command per
model, any provider, resume-aware, scores at the end) and `evaluation/run_free_tier.py`
(a quota-aware orchestrator for free-tier keys: prunes API-failure entries from
`preds.json` for retry, keeps genuine agent outcomes, sleeps and probes across daily
quota resets — built when this project ran on a free Gemini key before OpenRouter
credits were available; kept because it makes the benchmark runnable at $0).

Total model spend for all runs: ~$31 (OpenRouter) + $0 (Gemini free tier).

## Results

| model | n | mean score | resolved | easy | medium | hard |
|---|---|---|---|---|---|---|
| **glm-5.2** | 100 | **0.351** | **34 (34%)** | 0.56 | 0.10 | 0.18 |
| deepseek-v4-flash | 100 | 0.242 | 24 (24%) | 0.42 | 0.04 | 0.07 |
| qwen3-coder-30b | 100 | 0.037 | 3 (3%) | 0.05 | 0.03 | 0.00 |
| minimax-m2.5 *(partial)* | 29 | 0.103 | 3 (10%) | 0.23 | 0.00 | 0.00 |
| gemini-3.1-flash-lite, thinking-high *(partial)* | 8 | 0.356 | 2 (25%) | 0.50 | 0.28 | 0.00 |

(mean score in [0,1]; "resolved" = strict score 1.0; full per-module/exit-status
breakdowns in `results/summary.md`.)

**The benchmark is unsaturated and discriminative.** Scores span 0.037-0.351 across the full runs with a
consistent model ordering and a clean difficulty gradient in every run (easy > medium ≈
hard). No model approaches the ceiling; a frontier-class model (glm-5.2) leads at
0.351 mean / 34% resolved, roughly a third of the maximum.

**Where models fail differs, and the metric decomposes it.** Three gates compound into
"resolved": localize+fix the issue, do it within the step budget, and follow the
submission protocol.

- **deepseek-v4-flash** is precision-heavy: it submitted on only 27/100 tasks but 24 of
  those 27 were perfect (89% precision). Its bottleneck is wrapping up — 73 runs burned
  all 40 steps investigating without ever emitting the submission command.
- **qwen3-coder-30b** fails earlier: 29/100 runs collapsed on malformed tool calls
  (RepeatedFormatError) and only 10 submissions ever happened, 3 correct — a
  protocol-following weakness more than a coding one.
- **glm-5.2** has the highest submission rate (47/100, zero tool-call format errors) and 72% precision (34 of 47 submissions fully correct). It leads every difficulty tier yet still leaves two thirds of the benchmark unsolved.
- **minimax-m2.5** and **gemini-3.1-flash-lite** (partials) land between qwen and
  deepseek, consistent with their price/capability tier.

**Scoring pipeline integrity held throughout:** zero scorer exceptions across all runs; the only patch-application failures were three malformed qwen3-coder patches (itself a genuine agent failure mode). Every 0 traces to a real agent outcome: no submission, non-applying or wrong fix, or regression. The flash-lite partial excludes 7 runs lost to free-tier API rate-limit exhaustion (infrastructure, not agent, failures).

## Shortcomings

- **Single repository, single language.** Results measure "agent skill on tinygrad,"
  which is idiosyncratic (extremely terse style, heavy metaprogramming). Cross-repo
  generality is untested.
- **Problem-statement quality is bounded by the generator model.** Statements were
  written by `gemini-3.1-flash-lite` (the only model available on the key). Under- or
  over-specification directly shifts task difficulty; a subset was spot-checked manually
  but not exhaustively audited for leakage or unsatisfiability.
- **Statement/agent model overlap.** The same model family writes statements and solves
  tasks; a stronger statement generator (or human-audited statements) would be cleaner.
- **Protocol adherence is part of the score.** Runs that never emit the submission
  command score 0 even if the working-tree fix was correct (observed in smoke tests).
  This matches SWE-bench convention but mixes "can fix" with "can follow instructions".
- **Budget-tier models dominate the comparison.** The full runs use cheap models
  ($0.07-0.15/Mtok input); the single frontier-class datapoint (glm-5.2) covers only 50
  tasks. The harness runs any litellm model unchanged when more budget is available.
- **Recency bias.** Newest-first mining means tasks cluster in 2026 tinygrad; older eras
  of the codebase are unrepresented (deliberate, to keep one shared dependency layer).
- **arm64 image.** Built/run on Apple Silicon; x86-specific test files simply skip and
  were filtered out by validation, but reproducing on x86 requires rebuilding the image.

## Scaling & improvements

- **More repos:** the miner only assumes squash-merge history and a pytest suite; the
  same pipeline applies to any such repo (e.g. pytorch-lightning, httpx, pydantic),
  giving a multi-repo benchmark with per-repo dependency layers.
- **Bigger pool + difficulty calibration:** validation yield was ~71%; running the
  miner over the full 8.7k-PR window would give thousands of verified tasks, enabling
  held-out splits and IRT-style difficulty estimation from many model runs.
- **Statement audit loop:** generate statements with a strong model, then verify with an
  independent model that the statement (a) doesn't leak the fix and (b) contains every
  API fact the hidden tests need; regenerate on failure.
- **Task diversity:** the same infrastructure supports inverse tasks (write a failing
  test for a known bug, scored by fail-at-base/pass-at-fix) and performance tasks
  (scored by measured speedup) — both were designed but out-scoped here.
- **Paid-tier / multi-provider runs:** with any paid key, `run_benchmark.sh` evaluates
  frontier models directly; the report's comparison would then span model families
  rather than configurations.
